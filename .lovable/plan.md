
## Why you’re seeing 3 different numbers

They are not measuring the same thing.

### 1) Records = 6,538
This comes from the Records page in **Bookings view**.

Code path:
- `src/pages/Reports.tsx`
- `src/hooks/useReportsData.ts`

What it counts:
- rows from `bookings`
- filtered to `record_type = 'booking'`
- includes statuses like:
  - Pending Move-In
  - Moved In
  - Member Rejected
  - No Show
  - Cancelled
  - Postponed
  - Non Booking

Important detail:
`useReportsData` explicitly does:
- `eq('record_type', 'booking')`
- and then sets `summaryStats.total = totalCount`

So the **6,538** number is basically:
“all booking-type records in the system”
not
“actual converted bookings only”

That means it includes a lot of **Non Booking** rows if those rows were stored with `record_type = 'booking'`.

### 2) Dashboard “All Time Total Bookings” = 164
This is much narrower.

Code path:
- `src/pages/Dashboard.tsx`
- `src/hooks/useDashboardData.ts`
- `src/utils/dashboardCalculations.ts`

What it counts:
- first fetches rows from `bookings`
- then `calculateKPIData()` calls `filterActualBookings()`

That filter removes:
- `status === 'Non Booking'`
- `status === 'Research'`
- `recordType === 'research'`

So the dashboard KPI is not “all records”.
It is:
“actual booking records after excluding non-bookings and research”

That is why it can be much smaller.

There is also a bug risk here:
- `useDashboardData.ts` hard-stops after **10,000 rows**
- so “All Time” can silently truncate old data

But that truncation would not explain 164 vs 6,538 by itself. The main reason is the filter logic above.

### 3) Communication Insights → Booking Insights = 18,136
This is a different system entirely.

Code path:
- `src/components/call-insights/BookingInsightsTab.tsx`
- data comes from `member_insights.total_calls_analyzed`
- generated by `supabase/functions/analyze-member-insights/index.ts`

What that edge function counts:
- pulls from `bookings`
- requires `transcription_status = 'completed'`
- excludes only `status = 'Non Booking'`
- date-filtered by analysis period
- then counts rows with `booking_transcriptions.call_key_points`

Crucial bug:
it does **not** exclude:
- `status = 'Research'`
- `record_type = 'research'`

So Booking Insights is likely counting:
- real booking calls
- plus research records
- plus anything else that is not marked `Non Booking`

That explains why the number can be **18,136**, which is even larger than Records.

## So what the mess actually is

You currently have 3 different definitions:

```text
Records / Bookings tab
= all rows where record_type = 'booking'
= includes Non Booking rows

Dashboard / Total Bookings
= rows excluding Non Booking + Research + record_type=research
= “actual bookings” KPI

Communication Insights / Booking Insights
= saved analysis total_calls_analyzed
= excludes Non Booking only
= currently leaks Research records into booking analysis
```

## Most likely interpretation of your numbers

- **6,538** = all booking-type operational records
- **164** = true sales bookings after exclusions
- **18,136** = analyzed transcribed calls, likely contaminated by research rows and using a broader population than the dashboard

## Root causes in the code

### Root cause A — inconsistent definitions across pages
Each area defines “booking” differently:
- Reports uses `record_type`
- Dashboard uses `status + recordType`
- Booking Insights uses analysis pipeline rules

### Root cause B — Booking Insights edge function is too broad
In `supabase/functions/analyze-member-insights/index.ts`, the query excludes:
- `status != 'Non Booking'`

But it should also exclude:
- `status = 'Research'`
- `record_type = 'research'`

That is the biggest reason Communication Insights is inflated.

### Root cause C — Dashboard all-time fetch has a 10k cap
In `src/hooks/useDashboardData.ts`:
- pagination stops at 10,000 rows

That can make all-time dashboard numbers incomplete on large datasets.

## Fix plan

### 1. Standardize definitions
Define one canonical meaning for each metric:

- **Total Records**
  - all rows in `bookings`
- **Booking Records**
  - `record_type = 'booking'`
- **Actual Bookings**
  - booking records excluding `Non Booking` and `Research`
- **Research Records**
  - `record_type = 'research'`
- **Booking Insights Calls Analyzed**
  - only actual booking calls eligible for analysis

### 2. Fix Booking Insights source query
Update `analyze-member-insights` so booking analysis excludes:
- `status = 'Non Booking'`
- `status = 'Research'`
- `record_type = 'research'`

That will bring Communication Insights in line with actual booking analysis.

### 3. Fix Dashboard all-time data strategy
Replace the client-side all-time row fetch with server-side counts/aggregates, or at minimum remove/raise the 10k cap.
Recommended:
- server-side aggregate query for dashboard KPIs

### 4. Clarify labels in UI
Rename cards so users can tell what they’re seeing:
- Records page: “Total Booking-Type Records”
- Dashboard: “Actual Bookings”
- Communication Insights: “Calls Analyzed”

### 5. Add a visible breakdown card
Add a shared breakdown somewhere central:

```text
Total rows in bookings table
Booking-type records
Actual bookings
Non-booking calls
Research records
Analyzed booking calls
```

That will make discrepancies understandable instead of looking broken.

- Booking Insights analysis:
  - edge function filters only `.neq('status', 'Non Booking')`

### Recommended implementation order
1. Fix booking insights edge function filters
2. Fix dashboard all-time aggregation
3. Update UI labels
4. Add cross-page metric glossary/breakdown

Once aligned, the numbers should stop fighting each other:

- Records will still be the broad operational count
- Dashboard will show true actual bookings
- Communication Insights booking analysis will drop significantly and match booking-only scope
- Users will understand why each metric differs
